## Status

Accepted

## Context

この ToDo API は小規模な FastAPI マイクロサービスとして始まるが、開発効率と将来の保守性を両立できる実行基盤が必要である。既存リポジトリでは Python 3.14 が前提になっており、依存同期やテスト実行の手順も `uv` ベースで揃えつつある。

- **CTX-001**: 実行環境は、FastAPI と周辺ライブラリの利用可能性を維持しながら、開発者が迷わず再現できる必要がある。
- **CTX-002**: パッケージ管理は、依存解決・仮想環境・コマンド実行を一体で扱える仕組みが望ましい。
- **CTX-003**: この段階では、複雑な運用基盤よりも、軽量で一貫したローカル開発体験を優先する。

## Decision

Python ランタイムは `3.14` を採用し、パッケージ管理と実行コマンドは `uv` を正式採用する。依存同期、テスト、アプリ起動はすべて `uv` 経由で統一し、環境差分を最小化する。

- **DEC-001**: Python 3.14 を前提にする。
- **DEC-002**: 依存管理は `uv` を正規のツールチェーンとする。
- **DEC-003**: `pyproject.toml` を単一のメタデータ起点として運用する。

この選択により、開発者は同じコマンド体系で環境構築・依存同期・実行・テストを扱える。既存のリポジトリ方針とも整合し、余計な移行コストを増やさずに始められる。

## Consequences

### Positive

- **POS-001**: `uv` により依存同期と実行が簡潔になり、オンボーディング手順が短くなる。
- **POS-002**: `pyproject.toml` を中心に据えることで、設定の分散が減り、保守しやすくなる。
- **POS-003**: Python 3.14 を維持するため、将来の新しい言語機能や標準ライブラリ改善を早期に利用できる。
- **POS-004**: ローカル実行と CI のコマンド差分を抑えやすい。

### Negative

- **NEG-001**: Python 3.14 未対応の依存が見つかった場合、代替実装や依存見直しが必要になる。
- **NEG-002**: `uv` に不慣れな開発者には、最初の学習コストが発生する。
- **NEG-003**: 3.14 の利用は、周辺エコシステムの追随状況に影響されやすい。
- **NEG-004**: ツールチェーンを `uv` に寄せるため、他のパッケージ管理手段との二重運用は避ける必要がある。

## Alternatives Considered

### Python 3.12 へ下げる

- **ALT-001**: **説明**: 互換性重視で、より広い依存ライブラリ対応を優先する。
- **ALT-002**: **不採用理由**: 現時点では 3.14 前提の構成で問題なく進められ、将来性よりも現状の一貫性を優先した。

### Poetry を採用する

- **ALT-003**: **説明**: 一般的な Python パッケージ管理・仮想環境ツールとして Poetry を使う。
- **ALT-004**: **不採用理由**: 既存の作業方針と `uv` ベースの運用に合わせた方が、余計な移行と設定重複を避けられる。

### pip-tools を採用する

- **ALT-005**: **説明**: `pip-compile` を中心に固定依存を管理する。
- **ALT-006**: **不採用理由**: 依存解決と実行の体験が `uv` より分散しやすく、今回の小規模サービスにはやや重い。

## Implementation Notes

- **IMP-001**: 開発・CI では `uv sync --extra dev` を標準手順として扱う。
- **IMP-002**: 起動コマンドとテストコマンドは `uv run` 経由で統一する。
- **IMP-003**: `pyproject.toml` の依存定義を唯一の正とし、追加のロック戦略は増やしすぎない。
- **IMP-004**: Python バージョン差分が出た場合は、まず依存側の対応状況を確認してから下方修正を検討する。
